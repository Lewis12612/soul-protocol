## 一、发现的缺陷（来自 6/20 辅助 Agent 实例端日终执行）

### 缺陷 1：eod-pending 假完成（P0）

**现象**: 辅助 Agent 实例端守护进程触发日终，协议文本注入上下文，但 LLM 只回复"Medium协议执行完毕"，日终未执行。系统误判已完成，不再重试。

**根因**: `before-prompt-build.ts` 在协议文本**注入时**即更新 `last-eod.json`。此时 LLM 尚未执行任何日终动作。

**修复**: 
- 删除注入点的 `last-eod.json` 原子写入
- 改为标记 `eod-pending.json` 的 `injected: true` + `injected_at`，保留文件
- `last-eod.json` 仅在 LLM 实际执行日终时（`execute-protocol.ts` 的 `updateState("full", ...)`）更新
- `heartbeat.ts` 中 3 处同样模式同步修复

**影响范围**: `before-prompt-build.ts`, `heartbeat.ts`

### 缺陷 2：注入≠执行（P0）

**现象**: 协议的强制执行依赖"LLM 看到文本→主动执行"假设。心跳场景的短回复模板绕过了协议执行路径。

**修复**:
- `protocol.ts` 的 `renderProtocolPrompt` 中 Full 协议头部新增强制执行指令：
  ```
  ⚠️ 强制执行指令：检测到日终协议注入。必须在当前轮次中执行所有行动项，
  不得回复简略模板（HEARTBEAT_OK、Medium协议执行完毕 等）
  ```
- 这是提示词层面的增强，不是结构性修复。注入≠执行的根本解决方案仍在设计中。

**影响范围**: `protocol.ts`

### 缺陷 3：模板路径全部指向错误（P0）

**现象**: `dream-extract-prompt.md` 和 `full-workflow.md` 中引用的模板路径为 `templates/xxx.md`，实际文件位于 `templates/archive/xxx.md`。子 Agent 无法找到模板文件。

**修复**:
- `dream-extract-prompt.md`: 4 处 `templates/` → `templates/archive/`
- `full-workflow.md`: 14 处模板路径修正（含 `templates/consolidation/` 统一）

**影响范围**: `templates/archive/dream-extract-prompt.md`, `workflows/full-workflow.md`

### 缺陷 4：Spawn 子 Agent 超时 + 上下文污染（P1）

**现象**: 辅助 Agent 实例端（讯飞星辰 GLM-5.1）3 个 spawn 子 Agent 全部超时，work 归档输出昨日旧内容。

**根因（多因子）**:
1. `timeoutSeconds: 120` 对 GLM-5.1 过短（长思考链 + 云端算力紧张）
2. Spawn task 描述不包含 EXTRA 路径 → 子 Agent 无法定位数据源
3. Spawn task 不指定具体日期 → 子 Agent 在 EXTRA 不可用时回退到 `memory/*.md` 搜索，选中旧日期文件

**修复**:
- `sleepiness.json`: 新增 `spawn.timeoutSeconds` 配置段（归档 180s / 凝练 300s）
- `full-workflow.md`: Spawn task 增加 EXTRA 路径探针步骤（先读 `plugin.json` → `agentConfig.extraBasePath`）、日期歧义消除、`忽略 memory/*.md` 防护
- `full-workflow.md`: 所有路径从硬编码 `~/.openclaw/workspace/` 改为 `{workspace}` 占位符

**影响范围**: `config/sleepiness.json`, `workflows/full-workflow.md`

## 二、全量路径参数化

### 问题

V3.8.8-beta2 中多处硬编码了实例绑定的路径：
- `/tmp/openclaw-dialogue-logs`：extraBasePath 兜底
- `/home/openclaw`：HOME 变量兜底
- `{{dialogue_logs_path}}`：sleepiness-daemon.cjs 硬编码
- `~/.openclaw/workspace/skills/...`：full-workflow.md spawn task

这些路径在实例间迁移时全部需要手动修改。

### 参数化后的路径派生体系

```
所有路径从两个源派生：

源 1: os.homedir()（系统级，自动获取）
  → homeDir → {homeDir}/dialogue-logs（extraBasePath 默认值）

源 2: workspaceDir（插件配置，必填）
  → {workspaceDir}/memory/.heartbeat/
  → {workspaceDir}/skills/soul-protocol/templates/
  → {workspaceDir}/memory/memory-core/

覆盖: plugin.json → agentConfig.extraBasePath
       plugin.json → agentConfig.homeDir
       sleepiness.json → checkScriptsPath
```

| 文件 | 变更 | 说明 |

## 六、结构化日志系统

### 问题

V3.8.8-beta2 的日志系统极简陋：
- `watchdog.log`：纯文本，只有睡意轮询（"awake (0.21)"）
- 无执行追踪——看不到协议何时注入、LLM 是否执行、哪一步完成/失败
- 无结构化查询能力
- 日志无限增长，无剪枝

### 设计

```
memory/.heartbeat/logs/
├── plugin.log              ← 主日志（JSONL，所有模块共用）
└── archive/                ← 轮转归档
    └── plugin-{timestamp}.log
```

**JSONL 格式**:
```json
{"ts":"2026-06-21T08:51:00+08:00","mod":"eod","evt":"injected","lvl":"info","msg":"Full协议已注入","ctx":{"triggered_by":"watchdog","sleepiness":0.78}}
```

**模块标签**: `heartbeat` | `watchdog` | `eod` | `protocol` | `spawn` | `state` | `health`

**事件类型**: `injected` | `executed` | `step_done` | `step_fail` | `skipped` | `timeout` | `retry` | `error` | `health_check` | `state_change`

### 日志覆盖矩阵

| 事件点 | 模块 | 事件 | 文件 |
|--------|------|------|------|
| 心跳 dreaming 触发 | heartbeat | state_change | heartbeat.ts |
| 心跳 exhausted 触发 | heartbeat | state_change | heartbeat.ts |
| 对话中睡意耗尽 | heartbeat | state_change | heartbeat.ts |
| 心跳检查完成 | heartbeat | health_check | heartbeat.ts |
| EOD-pending 注入 | eod | injected | before-prompt-build.ts |
| LLM 调用 execute_protocol | protocol | executed | execute-protocol.ts |
| last-eod 状态更新 | eod | step_done | execute-protocol.ts |
| 守护进程睡意检查 | watchdog | health_check | watchdog.cjs |
| 守护进程触发日终 | watchdog | injected | watchdog.cjs |

### 剪枝

配置于 `sleepiness.json` → `logging.pruning`：

```json
"pruning": {
  "enabled": true,          // false → 完整保留所有日志
  "retentionDays": 30,      // 归档保留天数
  "maxArchiveFiles": 50     // 最多保留归档数
}
```

- `enabled: false` → `pruneArchives()` 直接返回，所有日志永久保留
- `enabled: true` → 每次轮转后触发，按时间和数量两个维度剪枝
- 默认开启，给希望完整记录的用户提供关闭选项

## 七、三层冗余架构实现

### 背景：注入≠执行

V3.7.3 时代心跳是**独立 agent turn**——LLM 在那个 turn 里只看到协议，只能执行协议。V3.8.8 改为 `prependSystemContext` 后，协议和对话共享同一个 token 窗口，LLM 在有活跃对话时"先回话"是天然更强的信号。

### A 层：协议独立 turn 注入

**原理**：eod-pending 触发时，写入 `protocol-turn-active.json` 标记。当前 turn 返回协议文本 + `⚠️ 当前轮为协议执行轮——对话消息将在下一轮处理`。下个 turn 检测到标记 → 清除 → 恢复正常对话。

**实现** (`before-prompt-build.ts`):
- L374-379: 检测 `protocol-turn-active.json` → 清除 → 恢复正常流程
- L478-520: eod-pending 消费时写入标记 + 独立轮头部

**防止卡死**：标记仅存活一个 turn，无论如何都会被清除。

### B 层：协议 = Tool Call 强制

**原理**：LLM 必须调用 `execute_protocol('full')` tool 作为接收协议的确认信号。

**实现**:
- `protocol.ts` L329: `🛠️ 强制 Tool 调用：必须首先调用 execute_protocol('full') tool`
- `execute-protocol.ts` L122-135: tool 被调用时记录 `protocol:confirmed` 日志 + 返回完整协议清单

### C 层：Watchdog 执行验证闭环

**原理**：外部进程（watchdog）独立于 LLM 循环，每 5 分钟验证 `injected_at` vs `last_eod_time`。

**实现** (`sleepiness-watchdog.cjs`):
- `verifyEodExecution()`: 检测 eod-pending 状态
  - `last_eod_time > injected_at` → 已执行 ✅ → 清理
  - 未执行 → retry_count++
    - retry=1: 标记 `injected=false` 允许重新注入
    - retry=2: 升级告警
    - retry≥3: 放弃重试，删除 eod-pending（防死循环）
- `checkLastEodStale()`: last-eod 超过 24h 未更新 → 健康告警

### P1：Spawn 超时自动化

**原理**：`sleepiness.json` 的 `spawn.timeoutSeconds` 在 `buildProtocol` 渲染时自动注入到每个 spawn_instruction 的 `timeoutSeconds` 参数中。

**实现** (`protocol.ts`):
- `loadSpawnTimeouts()`: 读取配置 + 默认值兜底
- 5 处 spawn_instruction 全部动态注入（daily:180/deep:180/work:180/default:120）

### 三层协同

```
Watchdog 检测 exhausted
  → 写入 eod-pending
  → Hook 消费 → A层接管（protocol-turn-active）
     → 注入协议 + B层强制 tool call
     → LLM 调用 execute_protocol('full') → last-eod 更新
  → C层（watchdog）检测到 last_eod_time > injected_at → 验证通过 ✅
  
  若 LLM 未执行:
  → C层 5min 后检测到未执行 → retry → 重新注入（A层再次接管）
  → retry≥3 → 放弃 + 告警
```

### 模拟测试结果（10/10 通过）

| 路径 | 测试内容 | 结果 |
|------|----------|:--:|
| A 层 | protocol-turn-active 创建→检测→清除 | ✅ |
| C 层 | 注入未执行 → retry → 标记 injected=false | ✅ |
| C 层 | retry 后执行 → last-eod 更新 → 验证闭环 | ✅ |
| 降级 | retry≥3 → 放弃 + 清理 | ✅ |
| B 层 | protocol.ts 含强制 tool call 指令 | ✅ |
| P1 | loadSpawnTimeouts 配置注入 | ✅ |

## 十、V3.9+ 展望

### 当前平台边界

beta3 已将 OpenClaw hook API 边界内的工程推到极限。以下为受限于架构/生态、留待未来的方向：

| 方向 | 限制 | 展望 |
|------|------|------|
| 协议结构性强制执行 | hook 只能 prepend 文本，不能拦截/调度 turn | 独立 agent 架构（脱离 OpenClaw）或 Gateway 级 turn 调度改造 |
| Provider 兼容矩阵 | 仅 DeepSeek V4 + GLM-5.1 可实测 | 社区反馈驱动的渐进兼容 |
| 单元测试体系 | 6300 行代码，测试基建成本高 | 核心模块优先（sleepiness/intent-resolver/dedup） |
| 部署摩擦 | 依赖 OpenClaw 插件体系 | Docker 化 + 独立部署脚本 |

### 设计哲学记录

- **身份层宪法不寻求自动化**：人格由用户定义，插件提供参考框架——这是产品定位，非功能缺失
- **不放弃 OpenClaw 生态**：当前阶段保留在 OpenClaw 平台的务实选择，独立架构为远期展望
